## 模型分配理由

| 模型 | 分配任务 | 理由 |
|------|----------|------|
| **GLM-5.1** | auto-01a + auto-01b | 检索任务需要**工具调用稳定性**，GLM的工具调用可靠性高；拆分后每个任务仅12-13组关键词，工具调用量可控 |
| **MiniMax-2.7** | auto-02a + auto-02b | 这4个方向**命中率低、需要语义扩展**，MiniMax在语义理解和发散检索上表现好；拆分后每组8组关键词，避免长链调用退化 |
| **Kimi-K2.6** | auto-03a + auto-03b + auto-03c + auto-04a + auto-04b | **长上下文**优势（200k+），适合处理大量论文的合并、去重、分类；输出任务需要**格式精确性**和**多文件操作**，Kimi的指令遵循能力强 |

> 为什么检索用GLM/MiniMax，整合用Kimi？
> - 检索是"广度"任务：需要调用工具、执行关键词、收集结果 → 工具稳定性优先
> - 整合是"深度"任务：需要阅读大量文本、做判断、格式化输出 → 上下文长度和指令遵循优先
>
> 为什么auto-02拆分为02a+02b？
> - 原16组关键词（48-64次工具调用）对MiniMax过长，后半段可能指令遵循退化
> - 拆分后每组8组关键词（24-32次调用），在可靠范围内

### ⚠️ 模型配置落地

WorkBuddy自动化模块**不支持按任务指定模型**，运行时使用全局当前模型。

**实际操作方案**：
- **方案A（采用）**：创建自动化时手动切换全局模型为该任务指定的模型（如创建01a时切GLM-5.1，创建03时切Kimi-K2.6），运行时模型取决于当时的全局设置
- **方案B**：若WorkBuddy后续支持per-task模型配置，再按原设计自动分配

> 每个任务文件头部已声明推荐模型（GLM-5.1/MiniMax-2.7/Kimi-K2.6），创建自动化时请按声明切换全局模型。若运行时全局模型与声明不一致，任务仍可执行，但效果可能偏离预期。

```
D:\.A硕士\.A干\每周文献调研(1)\每周文献调研\中间结果\
├── step01a-merged-YYYY-WXX.md     ← auto-01a输出

├── step03-classified-YYYY-WXX.md  ← auto-03a/03b输出

    └── journal-cache-YYYY-WXX/    ← 按周独立子文件夹
        ├── 10_1038_nbt_1234.md
        └── ...
```

### journal-cache 目录结构

**总文件夹**：`journal-cache/`（父文件夹，永久保留所有周数据）

**子文件夹**：`journal-cache/journal-cache-YYYY-WXX/`（每周一个独立子文件夹）

```
journal-cache/                              ← 父文件夹（所有旧周数据永久保留）
├── journal-cache-2026-W20/                ← W20缓存（永久保留）
│   ├── 10_1038_nbt_1234.md
│   └── ...
├── journal-cache-2026-W21/                ← W21缓存（永久保留）
│   └── ...
└── journal-cache-2026-W22/               ← W22缓存（永久保留）
    └── ...
```

**单个缓存文件格式**：
```markdown
# 期刊缓存 - [DOI]

| 字段 | 值 |
|------|---|
| 期刊名 | [期刊名] |
| JCR分区 | [Q1/Q2/Q3/Q4] |
| 影响因子 | [数值] |
| 更新日期 | [YYYY-MM-DD] |
```

**工作流程**：
1. auto-03c 读取 journal-cache/ 下所有子文件夹，加载历史缓存
2. 以DOI为键查找缓存；未命中则 WebSearch 查询
3. 查询结果存入 `journal-cache/journal-cache-YYYY-WXX/`（当周子文件夹）
4. 下周 auto-03c 自动复用历史缓存，避免重复查询

## 时间窗口自计算

每个自动化任务的prompt开头都包含：

```
## 时间窗口计算

请根据当前日期计算上周的检索时间窗口：
1. 上周一 = 今天往前推到今天所在的周一，再往前推7天
2. 上周日 = 上周一 + 6天
3. ISO周编号 = YYYY-WXX

示例：如果今天是2026-04-28（周二）
- 上周一 = 2026-04-20
- 上周日 = 2026-04-26
- ISO周编号 = 2026-W17
```

## 错误处理策略

| 场景 | 处理方式 |
|------|----------|
| 工具调用失败 | 重试2次，仍失败则跳过该工具，标注失败状态 |
| 某关键词零命中 | 正常记录，不扩展时间窗口 |
| 中间文件不存在（部分） | auto-03a允许1-2个文件缺失时继续，标注缺失 |
| 中间文件不存在（全部） | 停止执行，输出错误信息 |
| 整体结果极少（<5篇） | 记录警告，不自动扩展 |
| 核心方向零命中 | 标注"零命中"警告，建议手动扩展检索 |

## 数量控制

- 单方向最多保留 **8篇**
- 整合后：核心方向每方向最多 **5篇**，其他方向每方向最多 **3篇**
- 预印本：每方向最多 **3篇**
- 总数上限：**50篇**（按优先级截断）
- Obsidian笔记：仅 **⭐⭐⭐⭐及以上** 论文创建独立笔记

## 中间文件清理策略

**⚠️ WorkBuddy无法自动删除文件，不要在自动化任务中执行删除操作。**

若需清理旧中间文件，请手动执行：
1. 确认要删除的文件（3周前的step01/02/03文件）
2. 在文件管理器中手动删除
3. 不要依赖自动化任务中的删除指令

保留最近2周作为备份。**journal-cache 子文件夹永不过期删除**。

## 跨周补跑

1. 执行时检测上周输出文件是否存在
2. 若不存在 → 输出中标注"⚠️ 疑似跨周未执行"
3. 本次仅覆盖本周时间窗口，不自动补检缺失周次
4. **手动补跑**：手动触发对应周的自动化任务

## 执行监控（auto-04b自检）

1. 检查所有中间文件是否存在且非空
2. 检查Excel文件修改时间是否为当天
3. 检查周报文件存在且非空
4. 确认笔记数量与step03中⭐⭐⭐⭐及以上论文数量一致
5. 抽检3篇笔记的DOI/标题/核心发现与step03一致性
6. 自检结果写入周报「质量备注」

### B类 - 需人工介入的问题

| 问题 | 处理方式 |
|------|----------|
| 核心方向零命中 | 发送预警，建议手动扩展检索 |
| 中间文件缺失（1-2个） | 发送预警，标注缺失文件 |
| 中间文件缺失（3个及以上） | 发送严重预警，建议检查检索任务 |
| 周编号错乱/不连续 | 发送预警，建议手动确认 |

### 修复执行流程

1. 巡检阶段：检查所有文件完整性和正确性
2. 问题分类：分为A类（可自动修复）和B类（需人工介入）
3. 自动修复：对A类问题执行修复，记录修复原因和结果
4. 输出报告：汇总巡检结果、修复记录、剩余异常
5. 发送预警：对B类问题发送预警
